# M14 — Provider Verification / Embedding Namespace Hardening

## Goal

Make embedding/rerank provider changes safe, measurable, and non-silent. SourceBrief can use the default hashing/term-overlap providers for local alpha development, but production operators need explicit health evidence and retrieval must never silently mix vectors generated by incompatible providers/models/dimensions.

## Shipped behavior

### Provider health

`GET /provider-health` verifies the active provider configuration before operators rely on indexing/querying. It returns HTTP 503 when a provider probe fails so HTTP monitors do not treat a broken provider as healthy:

- embedding provider, model, dimensions, normalization mode, deployment id, and namespace
- embedding probe returns expected dimension
- normalized providers return an approximately unit-length vector
- rerank probe returns a score normalized into `[0, 1]`
- default hashing/term-overlap providers are explicitly marked `dev_quality=true`

Example response:

```json
{
  "status": "ok",
  "embedding": {
    "status": "ok",
    "provider": "hashing",
    "model": "sourcebrief-hashing-v1",
    "dimensions": 64,
    "normalized": true,
    "namespace": "hashing:sourcebrief-hashing-v1:d64:l2",
    "deployment_id": null,
    "dev_quality": true,
    "error": null
  },
  "rerank": {
    "status": "ok",
    "provider": "term-overlap",
    "model": "sourcebrief-term-overlap-v1",
    "score_range": [0.0, 1.0],
    "dev_quality": true,
    "error": null
  }
}
```

### Embedding namespace hardening

`chunk_embeddings` now stores:

- `provider`
- `model`
- `dimensions`
- `normalized`
- `namespace = provider:model:d{dimensions}:{l2|raw}[:dep-{deployment}]`

For real HTTP/OpenAI-compatible/HuggingFace/vLLM/SGLang providers, the namespace includes either `SOURCEBRIEF_EMBEDDING_DEPLOYMENT_ID` or a stable hash of `SOURCEBRIEF_EMBEDDING_ENDPOINT`. This prevents endpoint/deployment swaps with the same provider/model/dimension from silently mixing vectors.

Retrieval vector joins filter on the full active namespace. If an operator changes provider/model/dimension/normalization without reindexing, SourceBrief may still use lexical/graph/rerank candidates, but it will not silently use stale vectors from an incompatible namespace.

### Query diagnostics

Context packet responses include provider diagnostics:

```json
{
  "diagnostics": {
    "embedding_namespace": "hashing:sourcebrief-hashing-v1:d64:l2",
    "embedding_dimensions": 64,
    "embedding_normalized": true,
    "embedding_deployment_id": null,
    "vector_status": "ok",
    "matching_embedding_count": 3,
    "total_embedding_count": 3,
    "available_embedding_namespaces": ["hashing:sourcebrief-hashing-v1:d64:l2"],
    "rerank_score_range": [0.0, 1.0]
  }
}
```

`query_runs.metadata` stores the same namespace/dimension/normalization metadata for auditability. If active resources have embeddings but none match the current namespace, diagnostics report `vector_status="namespace_mismatch"` and list available namespaces instead of silently presenting semantic retrieval as healthy.

### Rerank normalization

Rerank scores are normalized/clamped into `[0, 1]`; non-finite values degrade to `0.0` instead of corrupting ranking math.

## Operator note

The default `hashing` embedding provider is deterministic and useful for local dev/test, but it is not semantic retrieval. Production deployments should configure a real embedding provider (`http`, `openai-compatible`, `huggingface`, `vllm`, or `sglang`) and use `/provider-health` before indexing large resources.

## Non-goals

- Hot-swapping vector dimensions without reindexing.
- Multiple simultaneous vector columns per chunk.
- Production semantic quality benchmarks; M18 owns the alpha evaluation gate.
